## 修复：选择设备模态框确认按钮无法按下（根本原因：内容区域高度限制）

**问题**：打开"选择设备"模态框后，确认按钮（"确定"）和取消按钮（"取消"）无法按下，甚至无法看到按钮。

**第一阶段诊断**：
- 问题表现：按钮看不到或无法点击
- 误诊：认为是 CSS 样式缺失
- 修复：添加了 `.btn` 类等样式
- 结果：问题依旧存在

**第二阶段诊断**：
- 进一步分析，添加内联样式
- 结果：问题仍未解决

**第三阶段 - 根本原因发现**：
- 真正的问题：`#deviceSelectContent` 有 `max-height: 300px` 和 `overflow-y: auto`
- 当设备列表较多时，内容被限制在 300px 高度内
- **按钮被挤到可见区域下方**，无法看到和点击
- 这是**模态框结构设计问题**，而非样式问题

**最终修复方案**：
将 `#deviceSelectContent` 的 `max-height` 从固定的 `300px` 改为 `calc(100vh - 400px)`：
- 这样可以动态计算最大高度，确保按钮始终可见
- 添加 `min-height: 100px` 防止内容区域过小
- 内容如果超出高度会滚动，但按钮始终在底部可见

**修复代码**（第 999-1002 行）：
```css
#deviceSelectContent {
    max-height: calc(100vh - 400px);  /* 从 300px 改为动态计算 */
    overflow-y: auto;
    min-height: 100px;               /* 新增 */
}
```

**修复文件**：
- `public/index.html`
- `public/index - 副本.html`

**验证方法**：
- **Ctrl+F5 硬刷新** 清除浏览器缓存
- 打开设备选择模态框
- 按钮应该始终在底部可见并可点击

## 修复：实时位置权限弹窗不出现

**问题**：点击"共享实时位置"后，浏览器既不弹出位置权限请求，也直接报"获取实时位置失败，请检查位置权限设置"。

**根本原因**：在 `shareRealTimeLocation()` 函数中，`navigator.geolocation.getCurrentPosition()` 调用之前有一个 `alert('开始共享实时位置，每30秒更新一次')` 弹窗（原第 16913 行）。`alert()` 会消耗/打断原始用户手势上下文，Chrome 等浏览器检测到手势已中断，导致后续的 geolocation 权限请求被静默拒绝，既不弹权限提示也不进入正常流程。

**修复方案**：将 `alert` 从 `getCurrentPosition` 调用前移至成功回调内部（position 已成功获取后再弹）。

**修复文件**：`public/index.html`（约第 16913-16930 行区域）

## 优化：位置共享 HTTP 禁用 + 拒绝后隐藏顶部按钮

**需求 1**：HTTP 非安全环境下，`showLocationOptions()` 弹出的模态框中两个位置按钮禁用，并显示黄色警告提示"请升级至 HTTPS 环境"。

**实现**：在 `showLocationOptions` 函数里用 `location.protocol` 检测，`localhost`/`127.0.0.1` 豁免，非安全环境注入 `httpsHint` 提示 div 并给按钮加 `pointer-events:none; opacity:0.45`。

**需求 2**：权限被拒（或任何获取失败）后，顶部红色"📍 取消实时定位"按钮不再显示。

**实现**：在 `shareRealTimeLocation` 的 error 回调中，除重置 `isSharingRealTimeLocation` 外，同步执行 `document.getElementById('locationShareStatus').style.display = 'none'` 并清 `realTimeLocationMessageId`；额外按 `error.code` 区分 1/2/3 给出更明确的提示文案。

**修复文件**：`public/index.html`（`showLocationOptions` 函数 + `shareRealTimeLocation` error 回调）

## 修复：选择设备模态框确认按钮无效

**问题**：从设置页面点"选择音频设备"打开模态框后，确认按钮按下没有任何反应。

**根本原因**：`confirmDeviceSelect()` 内部只有 `if (pendingCallAction)` 分支。`pendingCallAction` 仅在通话发起流程中赋值；从设置页面直接调用 `showDeviceSelectModal()` 时 `pendingCallAction` 为 null，导致确认按钮完全无效。

**修复方案**：
- `confirmDeviceSelect()` 新增 `else` 分支：`pendingCallAction` 为 null 时，将 `selectedVideoDeviceId` / `selectedAudioDeviceId` 保存到 `localStorage`（key: `preferredVideoDeviceId` / `preferredAudioDeviceId`），关闭模态框并提示"设备偏好已保存"
- `showDeviceSelectModal()`（设置入口）初始化时读取 localStorage 偏好，默认高亮上次保存的设备
- 通话发起时（视频/语音两处）也优先匹配 localStorage 偏好，找不到再 fallback 到第一个设备

**修复文件**：`public/index.html`（`confirmDeviceSelect`、`showDeviceSelectModal`、视频通话发起、语音通话发起四处）
